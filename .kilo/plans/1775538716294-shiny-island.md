# SQL Migration Plan for Supabase - Vendor & Tender System

## Overview

Migration untuk menambahkan sistem registrasi vendor, manajemen tender, monitoring proyek, dan notifikasi ke database LPrecast yang sudah ada.

## Prerequisites - Existing Database Analysis

### Existing Enums (DO NOT RECREATE)

- `stakeholder_type`: 'internal' | 'vendor' | 'client'
- `vendor_spk_status`: 'active' | 'completed'
- `vendor_payment_jenis`: 'dp' | 'term' | 'pelunasan'
- `customer_payment_termin`: 'dp' | 'term' | 'final'
- `letter_status`, `letter_action_type`, `meeting_status_enum`, `meeting_type_enum`, `kebutuhan_type`, `produk_type`, `sales_stage`

### Existing Tables

- `users`: id, email, nama, role_id, department_id, stakeholder_type, dll
- `clients`: id, nama, whatsapp, instagram_username, provinsi, kabupaten, kebutuhan, luasan, produk, status, tracking_source
- `projects`: id, name, customer_name, client_id, contract_value, created_at, updated_at
- `vendor_spk`: id, vendor_id (nullable), vendor_name, project_id, pekerjaan, nilai_spk, status
- `vendor_progress`: id, vendor_spk_id, tanggal, progress_percent, catatan, lampiran
- `vendor_payment`: id, vendor_spk_id, jenis_pembayaran, jumlah, tanggal_pembayaran, catatan, lampiran

### Existing Functions (for RLS)

- `current_user_id()`: returns uuid
- `is_vendor()`: returns boolean
- `is_client()`: returns boolean
- `is_internal_user()`: returns boolean

## Migration Steps

### SECTION 1: Create New Enums

```sql
-- Registration & Verification Enums
CREATE TYPE vendor_registration_status AS ENUM ('draft', 'submitted', 'under_review', 'approved', 'rejected', 'revision_requested');
CREATE TYPE document_verification_status AS ENUM ('pending', 'verified', 'rejected');
CREATE TYPE vendor_document_type AS ENUM ('ktp', 'npwp', 'nib', 'siup_sbu', 'company_profile');

-- Pricing Enums
CREATE TYPE cost_inclusion_type AS ENUM ('mobilisasi_demobilisasi', 'penginapan_tukang', 'biaya_pengiriman', 'biaya_langsir', 'instalasi', 'ppn');

-- Tender Enums
CREATE TYPE tender_status AS ENUM ('open', 'closed', 'cancelled', 'awarded');
CREATE TYPE tender_submission_status AS ENUM ('submitted', 'won', 'lost');

-- Project Enums
CREATE TYPE project_status AS ENUM ('draft', 'open', 'in_progress', 'completed', 'cancelled');
CREATE TYPE milestone_status AS ENUM ('pending', 'completed', 'overdue');

-- Payment Enums
CREATE TYPE payment_request_status AS ENUM ('pending', 'finance_verified', 'client_approved', 'paid', 'rejected');

-- Progress Enums
CREATE TYPE progress_status AS ENUM ('submitted', 'approved', 'rejected');

-- Notification Enums
CREATE TYPE notification_category AS ENUM ('vendor', 'tender', 'monitoring', 'payment', 'document', 'rab', 'general');

-- Vendor Profile Status Enum
CREATE TYPE vendor_profile_status AS ENUM ('active', 'suspended', 'blacklisted');
```

#### 9.1 vendor_registrations policies

```sql
-- Vendors can create new registrations
CREATE POLICY "vendors_can_insert_registrations" ON vendor_registrations
  FOR INSERT WITH CHECK (is_vendor() AND vendor_id = current_user_id());

-- Vendors can view their own registrations
CREATE POLICY "vendors_can_view_own_registrations" ON vendor_registrations
```

## Notes

1. **Foreign Key Constraints**: All foreign keys use `REFERENCES` to link to existing tables (users, projects, vendor_spk)
2. **Cascade Delete**: Tables with `ON DELETE CASCADE` ensure data integrity when parent records are deleted
3. **Nullable Fields**: vendor_id in vendor_spk, user_id in clients remain nullable as specified
4. **Indexes**: Strategic indexes on frequently queried columns (status, foreign keys)
5. **RLS Policies**: Using existing helper functions (is_vendor, is_client, is_internal_user, current_user_id)
6. **Timestamps**: All tables have `created_at` with DEFAULT now(), updateable tables have `updated_at`

## Execution Order

1. Create enums first
2. Alter existing tables
3. Create new tables (order matters due to foreign keys)
4. Enable RLS
5. Create RLS policies
6. Create triggers for updated_at

## Post-Migration Steps

After running migration:

1. Regenerate TypeScript types: `pnpm db:pull`
2. Test RLS policies with different user roles:
   - Test vendor registration flow (INSERT on vendor_registrations)
   - Test vendor can insert child records (company_info, contacts, etc.)
   - Test vendors can only see open tenders and their own submissions
   - Test internal users full access
   - Test notification creation by internal users
3. Verify triggers work correctly by updating records and checking updated_at timestamps
4. Consider adding unique constraints where business logic requires (e.g., unique constraint on vendor_products registration_id + name)
